IV) Problèmes dans les documents :

Voici divers éléments problématiques qui pourraient poser des problèmes si on fait des programmes utilisant ces documents :

  • Il n’y a pas de précisions sur le conditionnement de certains produits.
  • Les stocks sont insuffisants et parfois il n’y a pas d’information sur le stock disponible.
  • Les prix n’ont pas de devise définie (£, $).
  • On voit aussi que dans les fonctions dans le document Fournisseur, "Représentant(e)" et "Représentante" sont écrites différemment.
  • Pour le Brésil, Sao Paulo est écrit deux fois mais pas de la même manière.
  • Versailles est mal écrit (« î »), Paris aussi (la majuscule), ainsi que le Royaume-Uni et les États-Unis (erreur avec les tirets).
  • Il manque le numéro de rue pour L'Ancienne Auberge, ce qui pose divers problèmes.
  • Le format des numéros de téléphone n’est pas uniforme et certains numéros sont faux (par exemple, il manque des chiffres).
  • Il faudrait rajouter pour la livraison des informations supplémentaires, comme le moyen d'arriver, des numéros de bâtiment, etc.
  • La remise avec 8 décimaux (0.15000001) rend la lecture difficile et n'a aucun impact sur le prix.

V) Proposition de correction :

Pour corriger ces problèmes, il faudrait utiliser un algorithme de correction qui vérifie que les valeurs sont correctement rentrées, ou un code qui définit comment entrer les valeurs. De cette manière, les informations rentrées peuvent être utilisées par un programme sans qu’il y ait de bugs majeurs lors de l’exécution.
